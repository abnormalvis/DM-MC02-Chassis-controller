# DM-MC02 H723 底盘控制固件需求文档（草案 v0.3）

## 1. 文档目的与范围

本需求文档用于定义 DM-MC02（STM32H723）底盘控制固件的功能、实时性、安全策略与接口边界，目标是指导后续工程实现、联调与验收。

本固件在系统中扮演“底盘运动控制小脑”角色（定位为速度执行器）：

- 接收并解析遥控链路（CRSF/ELRS）控制指令。
- 接收上位机 RK3588 下发的离板控制指令（Offboard）。
- 执行双电机驱动输出与速度执行控制。
- 采集并上报 IMU/电流等关键状态给上位机（ROS2 节点）。

## 2. 系统总体架构

### 2.1 功能模块

- 输入链路：
  - CRSF/ELRS（UART）
    - USB + RS485 通信（第一阶段同时上线）
- 传感链路：
  - BMI088 IMU（SPI）
  - 双路电流采样：通过 C8T6 采集，H723 通过 CAN3 获取（CAN通信替代串口）
  - CAN 同步时间戳机制实现高精度延迟补偿
- 执行链路：
  - 4 路 PWM（20kHz）驱动两个电机
  - 状态指示（LED/蜂鸣器）
- 控制链路：
    - 运动控制源仲裁（CRSF 与 Offboard 控制互斥）
  - 电机控制（包含防溜坡前馈）

### 2.2 角色分工（已确认）

- RK3588：路径规划、速度目标生成、ROS2 系统集成、外部定位融合（GNSS）。
- H723：速度执行、执行器实时控制、IMU和电流采集、低层保护与状态回传。

## 3. 功能需求

### 3.1 CRSF/ELRS 接收与解析

1. 系统应支持通过 UART 接收 CRSF 数据帧。
2. 接收链路应采用 DMA + 空闲中断（IDLE）机制，避免高频数据造成 CPU 阻塞。
3. 解析器应完成：帧同步、长度检查、类型识别、CRC 校验、通道解包。
4. 输出统一的遥控输入结构体，至少包含：油门/转向/模式切换/急停。
5. 丢包或 CRC 异常应有错误计数并触发降级策略（见安全机制）。

### 3.2 运动控制源仲裁

1. 运动控制输入源包含：
    - RC 模式：CRSF/ELRS。
    - Offboard 模式：USB 或 RS485（USART2/USART3）。
2. RC 与 Offboard 的“运动控制指令”必须互斥，仅允许单一主控源生效。
3. USB/RS485 状态回传链路与 RC 独立，不受运动控制源互斥影响。
4. 模式切换必须满足防跳变策略：
    - 切换瞬间输出按斜坡限幅过渡。
    - 切换异常时进入刹车/安全态。

### 3.3 电机驱动 PWM 输出

1. 输出通道共 4 路：
    - 电机A：PWM1_A（正转）、PWM2_A（反转）
    - 电机B：PWM1_B（正转）、PWM2_B（反转）
2. PWM 频率固定为 20kHz。
3. 控制逻辑：
    - 正转：PWM1 > 0，PWM2 = 0
    - 反转：PWM1 = 0，PWM2 > 0
    - 刹车：PWM1 = 0，PWM2 = 0
4. 输出占空比范围统一归一化为 [0, 1]，并在底层映射到定时器计数值。
5. 应提供每路输出上限、爬升率限制和故障关断接口。

### 3.4 电流采样与电流环支撑

1. **架构变更**：DM-MC02 板上无 ADC 外设，电流采样由外接 C8T6 单片机完成。
2. H723 通过 **CAN3** (PD12=RX, PD13=TX) 与 C8T6 通信获取电流数据。
3. 通信协议：
   - CAN ID: 0x101 (C8T6→H723 电流数据)
   - 帧长度: 8 bytes
   - 数据格式: IA(2B) + IB(2B) + SEQ(1B) + STATUS(1B) + TIMESTAMP(2B)
   - 时间戳: CAN 控制器内置时间戳，单位 0.1ms
4. 同步机制：CAN 内置时间戳 + 序号，实现高精度延迟补偿
5. 波特率：1Mbps
6. 采样频率目标与主控制环一致（目标 1kHz）
7. 软件应提供：
   - 零点校准
   - 比例系数标定
   - 一阶低通滤波（可配置）
8. 当前阶段以”电流反馈监测 + 限流保护接口预留”为主；完整电流闭环可作为下一阶段需求。

### 3.5 IMU 采集与姿态估计

1. 板载 BMI088 通过 SPI 驱动。
2. 输出数据至少包含：三轴角速度、三轴加速度、姿态角（roll/pitch/yaw 或等价表示）。
3. 姿态更新频率目标不低于 1kHz（可根据总线与CPU负载降档）。
4. IMU 数据应作为以下功能输入：
    - 上位机状态回传
    - 防溜坡前馈计算
5. 允许后续接入 GNSS/里程计作为外部信息，不属于本阶段硬性实现。

### 3.6 USB/RS485 与上位机通信（第一阶段双链路上线）

1. 第一阶段同时实现 USB 与 RS485 通信。
2. RS485 使用 USART2 与 USART3 进行串口通信。
3. USB 使用独立 USB-DP 与 USB-DM 接口。
4. 协议层应统一，物理层无关，可在 USB 与 RS485 复用同一帧定义。
5. H723 需支持：
    - 接收 Offboard 控制指令（目标速度/角速度/模式）
    - 回传底盘关键状态（IMU、电流、模式、故障码）
6. 通信协议当前未最终冻结，先定义统一帧结构约束：
    - 帧头
    - 长度
    - 消息类型
    - 负载
    - CRC
7. 通信接口应具备心跳超时检测，超时进入安全态。

### 3.7 状态指示（LED/蜂鸣器）

1. 系统应至少区分以下状态并可视/可听提示：
    - 上电自检
    - 正常待机
    - RC 控制
    - Offboard 控制
    - 通信丢失/故障
    - 过流/保护触发
2. 提示模式需“非阻塞”实现，不得影响控制主环实时性。

## 4. 控制策略需求

### 4.1 主控制频率

- 控制主环目标频率：1kHz。
- 各任务允许分频运行，但运动输出更新时间不低于 1kHz。

### 4.2 防溜坡策略（当前版本，已确认）

1. 防溜坡采用纯前馈控制，不引入闭环速度环。
2. 前馈项基于 IMU 姿态信息计算（主要使用坡度分量）。
3. 前馈项应叠加 GNSS 定位/速度信息用于坡道场景补偿。
4. 前馈输出与驾驶指令叠加后，仍需经过限幅与斜坡约束。
5. 在 IMU 或 GNSS 信息异常时，应自动降级为仅驾驶指令输出并告警。

## 5. 实时性与任务划分（RTOS）

建议任务分层（优先级从高到低）：

1. 电机控制任务（1kHz，最高优先级）
2. CRSF 接收解析任务
3. USB 通信任务
4. RS485 通信任务
5. IMU 采集与解算任务
6. 状态机与故障管理任务
7. 指示灯/蜂鸣器任务

关键约束：

- CRSF 接收与 USB/RS485 通信并行执行。
- 任何通信任务不得阻塞控制主环。
- 所有跨任务数据需使用无锁环形队列或短临界区保护。

## 6. 安全机制（Failsafe）

1. 失联保护：
    - CRSF 超时或 Offboard 心跳超时触发失联。
2. 失联响应：
    - 默认策略为立即刹车（四路 PWM 归零，不做软刹车斜坡）。
3. 过流保护：
    - 任一电流通道超过阈值，进入限流或刹车保护态（阈值待标定）。
4. 模式冲突保护：
    - 若检测到控制源异常竞争，保持当前安全输出并上报故障。
5. 故障恢复：
    - 需满足“信号恢复稳定持续时间”后方可退出保护态。

## 7. 通信协议草案（v0）

### 7.1 下行（RK3588 -> H723）

- CMD_SET_MODE：设置模式（IDLE/RC/OFFBOARD）
- CMD_SET_CHASSIS_VEL：线速度、角速度目标
- CMD_ESTOP：急停
- CMD_PARAM_SET：参数下发（限幅、滤波系数、阈值）

### 7.2 上行（H723 -> RK3588）

- STA_HEARTBEAT：心跳、模式、故障字
- STA_IMU：角速度、加速度、姿态
- STA_MOTOR：两路电流、PWM占空比、保护状态
- STA_RC：RC链路状态与关键通道值

### 7.3 CRSF 链路定义（当前阶段）

- CRSF 作为 RC 控制输入链路，必须实现稳定接收与解析。
- CRSF 回传（Telemetry）第一阶段设为可选，不作为阶段验收阻塞项。
- 若启用 CRSF 回传，回传任务优先级低于电机控制与输入解析任务。

### 7.4 待确认

- 帧编码：二进制自定义协议（推荐）或直接复用现有协议栈。
- 字节序：小端（建议）。
- CRC 多项式：待定。
- 时间戳基准：MCU本地时基或上位机同步时基。

## 8. 接口与参数需求

### 8.1 关键可配置参数

- PWM 占空比上限
- 占空比变化斜率限制
- 电流零点/增益校准参数
- 过流阈值与恢复阈值
- 通信超时时间（RC/USB）
- 通信超时时间（RC/USB/RS485）
- 防溜坡前馈系数

### 8.2 调试接口

- 提供串口/USB 调试输出级别控制（ERROR/WARN/INFO/DEBUG）。
- 支持运行时读取关键状态与参数快照。

## 9. 验收标准（第一阶段）

1. CRSF：可稳定接收并解析，通道输出正确，异常帧不导致系统卡死。
2. PWM：4路20kHz输出稳定，正反转与刹车逻辑符合定义。
3. USB：可收发控制与状态帧，心跳超时可触发保护。
4. RS485：USART2/USART3 可收发控制与状态帧，心跳超时可触发保护。
5. IMU：可持续输出姿态相关数据并上报。
6. ADC：两路电流采集可用，映射到安培值误差满足标定预期。
7. RTOS：CRSF、USB、RS485 并行时控制环可稳定运行在目标频率附近。
8. 安全：失联与急停可在规定时间内切换到刹车态。
9. 实时性：主控制环频率误差容限当前阶段为 ±10%，第二阶段优化目标 ±5%。

## 10. 非功能需求

1. 稳定性：长时间运行无内存泄漏、无任务堆栈溢出。
2. 可维护性：模块化分层（驱动层、协议层、控制层、应用层）。
3. 可扩展性：通信物理层可从 USB 平滑扩展至 RS485。

## 11. 参数暂定值与冻结计划

以下内容用于解决当前“尚未完全敲定但需要先开工”的问题，采用“暂定值 + 冻结里程碑”机制。

### 11.1 通信与外设速率暂定值

1. CRSF UART 波特率：暂定 420000（若接收机不支持则回落 115200）。
2. RS485（USART2/USART3）波特率：暂定 921600，8N1。
3. USB：CDC 全速链路，应用层更新频率默认 100Hz 状态上报。
4. BMI088 SPI 速率：暂定 10MHz，后续按稳定性可提至 20MHz。
5. ADC：主环同步采样 1kHz，预留升频接口。

### 11.2 协议方向与职责暂定

1. CRSF：第一阶段强制实现接收解析；回传为可选增强项。
2. USB/RS485：统一协议帧，控制下行与状态上行均必须支持。
3. 上位机控制源在 Offboard 模式下可选择 USB 或 RS485 任一链路，冲突时按最近有效帧优先。

### 11.3 PWM 关键参数暂定值

1. PWM 频率：20kHz（固定）。
2. 定时器计数精度：根据时钟与分频计算，优先保证频率正确。
3. 占空比上限：初始 0.90。
4. 占空比爬升限制：每 1ms 最大变化 0.02（可参数化）。
5. 刹车命令：四路占空比在单周期内归零。

### 11.4 参数冻结里程碑

1. M1 联通测试：冻结 UART/SPI 速率。
2. M2 闭环联调：冻结协议帧字段与 CRC。
3. M3 场地测试：冻结 PWM 上限、斜坡、过流阈值。

## 12. 已知风险与后续计划

### 12.1 已知风险

- USB 通信协议尚未冻结，可能影响上位机联调节奏。
- 仅靠 IMU 前馈防溜坡，在低附着/强扰动场景效果有限。
- 若缺少轮速/编码器，速度估计与里程计能力受限。

### 12.2 第二阶段建议

- 引入轮速反馈，形成速度闭环。
- 引入 GNSS/里程计融合，提高坡道工况稳定性。
- 补全电流环闭环控制与保护策略。

## 13. 已确认决策清单

1. H723 只做速度执行器，不承担上层路径规划。
2. 第一阶段 USB 与 RS485 同时上线。
3. 失联策略采用立即刹车。
4. 防溜坡采用 IMU 姿态前馈 + GNSS 定位信息。
5. 主环频率误差容限先定 ±10%，后续优化到 ±5%。

## 14. 当前待定项（不阻塞开工）

1. 串口与 SPI 的最终稳定速率（按 11.1 暂定值先实现）。
2. CRSF Telemetry 是否纳入第一阶段验收（当前为可选）。
3. USB/RS485 协议字段与 CRC 最终版本。
4. PWM 斜坡和上限参数在实车负载下的最终标定值。

## 15. 实现前接口清单（v0）

本节用于指导第一阶段代码落地，字段、ID、单位、超时与状态码先按本版本实现。

### 15.1 USB/RS485 统一二进制帧格式

#### 15.1.1 帧结构

固定帧头 + 变长载荷，适用于 USB CDC 与 RS485（USART2/USART3）。

| 字段 | 类型 | 字节数 | 说明 |
|---|---:|---:|---|
| SOF | uint16 | 2 | 帧头，固定 0x55AA |
| VER | uint8 | 1 | 协议版本，v0 取 0x01 |
| MSG_ID | uint16 | 2 | 消息类型 |
| SEQ | uint16 | 2 | 发送序号，循环递增 |
| TS_MS | uint32 | 4 | 时间戳，毫秒计数 |
| LEN | uint16 | 2 | Payload 字节长度 |
| PAYLOAD | bytes | N | 数据体 |
| CRC16 | uint16 | 2 | CRC16-CCITT-FALSE（暂定） |

约束：

1. 字节序采用小端。
2. 单帧最大载荷建议不超过 128 字节。
3. 接收端必须先校验 SOF、LEN、CRC，再进入业务处理。
4. USB 与 RS485 复用同一解析器，仅链路驱动层不同。

#### 15.1.2 消息 ID 规划（v0）

下行（RK3588 -> H723）：

| 消息名 | MSG_ID | 周期 | 必需 |
|---|---:|---:|---|
| CMD_HEARTBEAT | 0x0101 | 20ms | 是 |
| CMD_SET_MODE | 0x0102 | 事件触发 | 是 |
| CMD_SET_CHASSIS_VEL | 0x0103 | 10ms | 是 |
| CMD_ESTOP | 0x0104 | 事件触发 | 是 |
| CMD_PARAM_SET | 0x0105 | 事件触发 | 否 |
| CMD_PARAM_GET | 0x0106 | 事件触发 | 否 |

上行（H723 -> RK3588）：

| 消息名 | MSG_ID | 周期 | 必需 |
|---|---:|---:|---|
| STA_HEARTBEAT | 0x0201 | 100ms | 是 |
| STA_IMU | 0x0202 | 10ms | 是 |
| STA_MOTOR | 0x0203 | 10ms | 是 |
| STA_RC | 0x0204 | 20ms | 是 |
| STA_FAULT | 0x0205 | 事件触发+100ms重发 | 是 |
| STA_PARAM | 0x0206 | 应答帧 | 否 |

#### 15.1.3 关键载荷定义

CMD_SET_MODE（0x0102）：

| 字段 | 类型 | 单位 | 说明 |
|---|---|---|---|
| mode | uint8 | - | 0:IDLE, 1:RC, 2:OFFBOARD |
| source | uint8 | - | 1:USB, 2:RS485 |

CMD_SET_CHASSIS_VEL（0x0103）：

| 字段 | 类型 | 单位 | 说明 |
|---|---|---|---|
| vx_mps | float32 | m/s | 线速度目标 |
| wz_rps | float32 | rad/s | 角速度目标 |
| reserved | uint16 | - | 保留 |

STA_IMU（0x0202）：

| 字段 | 类型 | 单位 |
|---|---|---|
| gx, gy, gz | float32 x3 | rad/s |
| ax, ay, az | float32 x3 | m/s^2 |
| roll, pitch, yaw | float32 x3 | rad |

STA_MOTOR（0x0203）：

| 字段 | 类型 | 单位 | 说明 |
|---|---|---|---|
| i_a, i_b | float32 x2 | A | 两路电流 |
| duty_a_fwd, duty_a_rev | float32 x2 | [0,1] | 电机A输出 |
| duty_b_fwd, duty_b_rev | float32 x2 | [0,1] | 电机B输出 |
| brake_active | uint8 | - | 0/1 |

#### 15.1.4 超时与丢链策略

1. Offboard 心跳超时：100ms（连续超时即失联）。
2. CRSF 超时：100ms（可按接收机刷新率调整）。
3. 失联后动作：立即刹车（四路 PWM 归零）。
4. 恢复条件：链路连续稳定 500ms 后允许退出保护态（暂定值）。

#### 15.1.5 状态码与故障码

状态码（mode_state）：

- 0: BOOT
- 1: IDLE
- 2: RC_ACTIVE
- 3: OFFBOARD_ACTIVE
- 4: BRAKE_PROTECT

故障码（fault_bits，uint32）：

- bit0: RC_TIMEOUT
- bit1: OFFBOARD_TIMEOUT
- bit2: MODE_CONFLICT
- bit3: ADC_OVERCURRENT
- bit4: IMU_FAULT
- bit5: GNSS_FAULT
- bit6: PARAM_CRC_ERR
- bit7: PROTOCOL_ERR

#### 15.1.6 基于 ROS底板协议.pdf 的 USB 协议补充（兼容层）

参考资料：项目文档 [projects/dm-mc02-bottom-core/docs/ROS底板协议.pdf](projects/dm-mc02-bottom-core/docs/ROS底板协议.pdf)。

该 PDF 定义了一套固定 12 字节帧格式，建议在第一阶段实现“兼容解析/发送层”，便于与现有上位机或历史节点互通。

Legacy 12 字节帧格式（暂定命名 legacy_v1）：

| 字段 | 字节数 | 说明 |
|---|---:|---|
| SOF | 1 | 固定 0xFC |
| FUNC | 1 | 功能码（0x01~0x08） |
| D0..D7 | 8 | 载荷字节 |
| XOR | 1 | 前 10 字节按位异或 |
| EOF | 1 | 固定 0xDF |

约束：

1. 接收端先校验 SOF/EOF，再校验 XOR。
2. 若校验失败，计入协议错误计数并丢弃帧。
3. legacy_v1 与 15.1.1 的统一协议可并存，由链路配置决定启用模式。

PDF 功能码语义整理：

| FUNC | 方向 | 语义 | 对应 v0 MSG_ID |
|---:|---|---|---|
| 0x01 | H723 -> 上位机 | 电池电压上报 | STA_MOTOR(0x0203) 扩展字段 battery_mv |
| 0x02 | H723 -> 上位机 | 四轮编码器值 A/B/C/D | 新增 STA_ENCODER(0x0207) |
| 0x03 | H723 -> 上位机 | 陀螺仪数据 XYZ（文档注释：值/10） | STA_IMU(0x0202) |
| 0x04 | H723 -> 上位机 | 加速度数据 XYZ（文档注释：值/10） | STA_IMU(0x0202) |
| 0x05 | H723 -> 上位机 | 欧拉角（俯仰/横滚/航向，值/10） | STA_IMU(0x0202) |
| 0x06 | 上位机 -> H723 | 四电机期望转速 A/B/C/D | CMD_SET_WHEEL_RPM(0x0107) |
| 0x07 | 上位机 -> H723 | PID 参数 P/I/D | CMD_SET_PID(0x0108) |
| 0x08 | 上位机 -> H723 | 舵机角度控制 | CMD_SET_SERVO(0x0109) |

补充说明：

1. 当前底盘项目为双电机四路 PWM 架构，legacy 的 A/B/C/D 轮速字段保留为兼容接口。
2. 在本项目中建议映射为：A/B 生效，C/D 预留（默认回 0 或镜像 A/B，按联调约定）。
3. 若上位机已固定发送 0x06 指令，可在协议适配层直接转换为本项目 vx/wz 或双电机目标值。

#### 15.1.8 USB 协议实现建议（结合 PDF）

1. USB 驱动层支持两种协议模式：
    - unified_v0：15.1.1 定义的可扩展帧。
    - legacy_v1：PDF 的 12 字节 FC...DF 帧。
2. 协议适配层将 legacy_v1 映射到统一内部消息总线，业务层只处理统一消息结构。
3. 联调顺序建议：先打通 legacy_v1（便于快速对接现有上位机），再切到 unified_v0。
4. 若需严格与 PDF 保持一致，XOR 校验作为 legacy_v1 唯一校验；unified_v0 继续使用 CRC16。

### 15.2 CRSF 输入映射与模式仲裁状态机

#### 15.2.1 CRSF 输入映射（v0）

建议通道映射：

| 功能 | CRSF通道 | 范围 | 输出归一化 |
|---|---:|---|---|
| 油门 | CH1 | 172~1811 | -1.0~1.0 |
| 转向 | CH2 | 172~1811 | -1.0~1.0 |
| 模式拨杆 | CH5 | 三段 | 0:IDLE,1:RC,2:OFFBOARD请求 |
| 急停拨杆 | CH6 | 两段 | 0:正常,1:急停 |

注：具体通道号可按遥控器配置调整，但代码接口保持语义不变。

#### 15.2.2 控制源仲裁规则

1. RC 与 Offboard 的运动控制互斥。
2. 当 mode=RC_ACTIVE 时，仅 CRSF 速度指令生效。
3. 当 mode=OFFBOARD_ACTIVE 时，仅 USB/RS485 最近有效帧生效。
4. 若 USB 与 RS485 同时下发 Offboard 指令，按“最近有效帧优先”。
5. 任一急停触发（RC 或上位机 CMD_ESTOP）均立即刹车。

#### 15.2.3 状态机定义

状态集合：BOOT、IDLE、RC_ACTIVE、OFFBOARD_ACTIVE、BRAKE_PROTECT。

迁移条件：

1. BOOT -> IDLE：初始化完成且自检通过。
2. IDLE -> RC_ACTIVE：RC有效且模式请求为RC。
3. IDLE -> OFFBOARD_ACTIVE：Offboard心跳有效且模式请求为OFFBOARD。
4. RC_ACTIVE <-> OFFBOARD_ACTIVE：满足请求且目标链路连续稳定 100ms。
5. 任意状态 -> BRAKE_PROTECT：急停、失联、模式冲突、过流。
6. BRAKE_PROTECT -> IDLE：故障解除且稳定 500ms。

#### 15.3.2 任务周期与预算（v0）

| 任务 | 周期 | 预算 | 备注 |
|---|---:|---:|---|
| MotorControlTask | 1ms | <200us | 计算前馈、限幅、下发PWM |
| AdcSampleTask | 1ms | <100us | 电流采样与滤波 |
| ImuTask | 1ms | <250us | SPI读取与姿态更新 |
| CrsfParseTask | 1ms轮询/中断触发 | <100us | DMA+IDLE取包 |
| UsbCommTask | 2ms | <150us | 协议收发 |
| Rs485CommTask | 2ms | <150us | USART2/3 协议收发 |
| SafetyTask | 5ms | <80us | 超时、冲突、故障迁移 |
| IndicatorTask | 20ms | <50us | LED/蜂鸣器非阻塞更新 |

#### 15.3.3 时序要求

1. control_step_1khz 在每个 1ms 周期内只能执行一次。
2. 任一通信任务不得持有互斥锁超过 50us（建议）。
3. 失联触发到 pwm_brake_immediate 的延迟目标小于 2ms。
4. 第一阶段统计窗口内主环频率误差需满足 ±10%。
