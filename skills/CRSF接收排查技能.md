# CRSF 接收排查技能

适用场景：STM32H7 平台上的 CRSF / ELRS 接收、DMA 空闲中断解析、D-Cache、UART 错误、链路统计解码问题。

## 1. 目标

这份技能文档用于快速定位 CRSF 接收链路中的三类问题：

- 物理层 / UART 层错误，例如 `fe`、`ne`、`ore`、波特率不匹配。
- DMA / 内存布局错误，例如缓冲区放到 DTCM、D-Cache 未对齐、缓冲区与堆栈冲突。
- 协议解析错误，例如帧长度判断错误、link-stats 帧解码条件写错、统计口径混淆。

## 2. 已验证的排查流程

### 2.1 先看现象，不先改逻辑

优先确认下面几项：

- `conn` 是否为 1。
- `uart_tot` 是否为 0。
- `pe/fe/ne/ore` 是否有增长。
- `LQ` 是否能稳定显示 99~100。
- `frame` 是否持续增长。
- `parse` 是否真的是解析失败，还是统计口径问题。

如果 `conn=1`、`uart_tot=0`、`LQ=100`，通常说明链路已通，问题多半在统计或解析分支，不是硬件失联。

### 2.2 按顺序排查

1. **内存布局**
   - DMA 缓冲必须放在 AXI SRAM，不能放 DTCM。
   - STM32H7 上 DMA 缓冲不要贴近栈顶，避免链接地址与 STACK 冲突。
   - 需要时把缓冲固定到 `0x24000000` 一类安全地址。

2. **D-Cache 兼容性**
   - DMA 读写缓冲要按 32 字节对齐。
   - CPU 读 DMA 区前，先做 D-Cache invalidate。

3. **UART 参数**
   - 确认波特率和接收机一致。
   - 在这套工程里，USART3 使用过 420000 的 CRSF 标准值。
   - 波特率不一致时，最容易出现 `fe/ne/ore`。

4. **协议长度与类型**
   - link-stats 帧总长是 14 字节，不是 15。
   - heartbeat 帧很短，不能按 link-stats 的条件去卡死。
   - 解析前先分清：RC 帧、link-stats 帧、heartbeat 帧、其他帧。

5. **DMA 重启时机**
   - RX 回调里应尽快重启 DMA。
   - 不要把耗时解析放在重启 DMA 之前。
   - 若出现 ORE，优先缩短回调中的处理路径。

## 3. 这次排查中真实遇到的问题

### 3.1 L6971E 链接错误

症状：CRSF 缓冲区放在栈区附近，和 STACK 发生冲突。

结论：DMA 缓冲应放在 AXI SRAM 起始区，避免与堆栈下行增长碰撞。

### 3.2 ADC HardFault

症状：DMA 访问非法地址，伴随 BFAR 异常。

结论：DMA 缓冲不能放 DTCM；同时要考虑 D-Cache 行对齐。

### 3.3 USART3 波特率错误

症状：`fe/ne/ore` 出现，`uart_tot` 不为 0。

结论：USART3 波特率需要与 CRSF 设备完全同步。

### 3.4 link-stats 解码失败

症状：`LQ=0`，但 RC 通道部分能动。

结论：link-stats 帧长度判断写错，正确总长为 14 字节。

### 3.5 ORE 过载

症状：接收过程中偶发 overrun。

结论：应先恢复 DMA 接收，再进行本地缓存解析。

### 3.6 统计口径误判

症状：`sync_err` 持续增长，但 `err=0`、`parse=0`、`uart_tot=0`。

结论：`sync_err` 如果按“扫描时跳过字节数”统计，就不能当成错误数看。应拆成：

- `sync_loss_events`：失去同步的事件数。
- `sync_skip_bytes`：扫描过程中跳过的字节数。

## 4. 判读规则

### 4.1 这是正常还是异常

正常特征：

- `conn=1`
- `uart_tot=0`
- `LQ=99` 或 `100`
- `RC` 数值持续变化
- `parse=0`

异常特征：

- `uart_tot` 增长，尤其是 `fe/ne/ore`。
- `LQ` 长期为 0。
- `conn=0` 且 `idle_ms` 持续变大。
- `RC` 数值卡死不动。

### 4.2 统计值怎么解读

- `parse=0` 但 `sync_skip` 很大：通常不是错误，而是扫描原始缓冲时跳过了非同步字节。
- `crc_error` 增长：多半是真数据损坏、波特率错、DMA 采样窗口错。
- `rc_decode_fail` 增长：RC 帧进来了，但通道解包规则不匹配。
- `len_err` 增长：帧类型长度判断有问题。

## 5. 最小化验证清单

遇到新的 CRSF 异常时，按以下顺序做最低成本验证：

1. 看 `uart_tot` 和 `fe/ne/ore`。
2. 看 `conn`、`frame`、`parse` 是否同步变化。
3. 看 `LQ` 是否从 0 恢复到 99~100。
4. 看 RC 通道是否实际在变化。
5. 若只有 `sync_skip` 很大，先不要把它当故障。

## 6. 推荐实现习惯

- DMA RX 缓冲固定在 AXI SRAM。
- 缓冲区按 32 字节对齐。
- RX 回调只做：拷贝、重启 DMA、轻量解析。
- 帧类型检查要分开，不要把所有非 RC 帧都算成错误。
- 调试输出要区分“错误”和“扫描过程量”。

## 7. 这次问题的最终结论

CRSF 接收链路本身已稳定：

- UART 没有错误。
- DMA 没有过载。
- `LQ` 已正确解码。
- RC 通道可以连续更新。

当前最重要的经验是：**不要把同步扫描过程量误当成解析错误**。如果统计口径不清晰，调试日志会比硬件更早制造假象。
